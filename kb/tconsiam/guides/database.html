Overview

TCONSIAM uses SQLite 3 with WAL (Write-Ahead Logging) mode, managed through Drizzle ORM for type-safe database access. All data lives in a single file on the VPS.

Why SQLite? Single-server deployment with 1-5 concurrent users. WAL mode provides ACID compliance, concurrent reads, and zero DevOps overhead. Backups are simple file copies.

Drizzle ORM Usage

Schema Definition Example

// api/src/db/schema.ts
import { sqliteTable, integer, text } from 'drizzle-orm/sqlite-core';

export const products = sqliteTable('products', {
  id: integer('id').primaryKey(),
  sku: text('sku').notNull(),
  name: text('name').notNull(),
  description: text('description'),
  unitCost: integer('unit_cost').notNull(),
  sellingPrice: integer('selling_price').notNull(),
  companyId: integer('company_id').notNull(),
});

Query Examples

// SELECT with filter
const items = await db.select()
  .from(products)
  .where(eq(products.companyId, user.companyId));

// INSERT
await db.insert(products).values({
  sku: 'LAB-001', name: 'Labubu Mini',
  unitCost: 250, sellingPrice: 590,
  companyId: user.companyId,
});

// UPDATE
await db.update(products)
  .set({ sellingPrice: 690 })
  .where(eq(products.id, productId));

Migrations

27+ versioned migrations in api/src/db/migrate.ts. Migrations run automatically on server startup.

Adding a New Table

Step 1: Define schema in api/src/db/schema.ts
  export const myTable = sqliteTable('my_table', { ... });

Step 2: Add migration SQL in api/src/db/migrate.ts
  CREATE TABLE IF NOT EXISTS my_table (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now'))
  );

Step 3: Restart API — migration runs automatically
  systemctl restart tconsiam-api
Important: Always use CREATE TABLE IF NOT EXISTS to make migrations idempotent. Never drop tables in production.

WAL Mode

WAL (Write-Ahead Logging) is enabled at database initialization in api/src/db/index.ts.

Benefits

  • Concurrent reads — Multiple readers don't block each other
  • Non-blocking writes — Readers continue during writes
  • Crash safety — ACID compliant, auto-recovery on restart
  • Better performance — Fewer disk syncs than journal mode

WAL Files

~/.config/tconsiam/db/
  tconsiam.db          # Main database
  tconsiam.db-wal      # Write-ahead log (auto-managed)
  tconsiam.db-shm      # Shared memory index (auto-managed)
Never delete .db-wal or .db-shm files while the API is running. They contain uncommitted data. Stop the API first if you need to copy the database.

Backup & Restore

Safe Backup (API running)

# Use SQLite backup command (safe with WAL)
sqlite3 ~/.config/tconsiam/db/tconsiam.db ".backup /tmp/tconsiam-backup.db"

# Or stop API first for file copy
systemctl stop tconsiam-api
cp ~/.config/tconsiam/db/tconsiam.db /backup/tconsiam-$(date +%Y%m%d).db
systemctl start tconsiam-api

Restore

systemctl stop tconsiam-api
cp /backup/tconsiam-20260409.db ~/.config/tconsiam/db/tconsiam.db
systemctl start tconsiam-api
Principle: Nothing is Deleted. Always keep backups before any schema change or data migration.

Troubleshooting

Database locked

  • Check for long-running queries or zombie processes
  • Verify WAL mode is enabled: sqlite3 tconsiam.db "PRAGMA journal_mode;"
  • Restart API if needed: systemctl restart tconsiam-api

Migration fails on startup

  • Check API logs: journalctl -u tconsiam-api -n 50
  • Verify SQL syntax in api/src/db/migrate.ts
  • Backup DB before retrying

Data not appearing

  • Check companyId filter — all queries are scoped to user's company
  • Verify JWT token is valid and contains correct userId
