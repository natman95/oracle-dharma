Key Design Decisions

Why SQLite (not PostgreSQL)?

Rationale: Single-server deployment with low concurrency (1-5 concurrent users). SQLite WAL mode provides ACID compliance, concurrent reads, and zero DevOps overhead. Perfect for TCONSIAM's scale.
  • Zero configuration — just a file
  • WAL mode enables concurrent reads while writing
  • Backups are simple file copies
  • Zero external dependencies (no PostgreSQL server to manage)
  • Perfect for single-VPS deployments

Why Bun + Hono (not Node.js + Express)?

Rationale: Bun is a modern JavaScript runtime with native TypeScript, faster startup, lower memory footprint. Hono is lightweight, has native Bun support, and minimal framework overhead.
  • Bun startup: ~30ms vs Node.js: ~200ms
  • Memory: 40MB vs Node.js: 80MB
  • Native TypeScript compilation (no tsc needed)
  • Hono: 20KB vs Express: 50KB (lighter)
  • Better for small-medium server loads

Why Drizzle ORM (not Prisma or raw SQL)?

Rationale: Drizzle is lightweight, type-safe, and SQLite-optimized. Zero runtime overhead, TypeScript-first design.
  • Type-safe queries at compile time
  • Lightweight (no runtime dependency injection)
  • Works with SQLite, PostgreSQL, MySQL
  • Raw SQL when needed (via sql tag)
  • Easy migrations

Why Hash-Based Routing (SPA)?

Rationale: Single-Page App with hash routing (#/) requires no server-side routing. Works with static hosting, easy to reason about state.
  • No server-side routing needed — static hosting sufficient
  • Client-side navigation is instant
  • Zustand store persists state in-memory
  • Works offline after first load

Why Computed Stock (not Stored Column)?

Rationale: Stock is computed as SUM(quantity) from inventory_movements. This ensures the audit trail is the source of truth.
  • No duplication — single source of truth
  • Automatic consistency (can't get out of sync)
  • Full movement history preserved for auditing
  • Easy to fix with a re-sum if needed

Request-Response Data Flow

Typical API Call (with Authentication)

1. Browser User Action
   └─ React component dispatches action
      (e.g., ProductsView.tsx: "fetch products")

2. Client Library (api.ts)
   └─ apiFetch(endpoint, options)
      ├─ Read JWT token from Zustand store
      ├─ Set Authorization header: "Bearer <token>"
      └─ Fetch /api/products (proxied to :3457)

3. Nginx (port 8890)
   └─ Route /api/* → http://localhost:3457/
      (static files served directly)

4. Hono API Server (:3457)
   ├─ Match route: GET /api/products
   ├─ Run middleware:
   │  ├─ authMiddleware (extract JWT)
   │  ├─ verifyToken (check expiry, signature)
   │  ├─ checkPermission (role-based guard)
   │  └─ requestLogger (log to stdout)
   │
   ├─ Execute handler: async (c) => {...}
   │  ├─ Extract userId from JWT
   │  ├─ Query database:
   │  │  db.select().from(productsTable)
   │  │    .where(eq(productsTable.companyId, userId.companyId))
   │  │
   │  ├─ Apply sorting/filtering/pagination
   │  └─ Build response: { products: [...], count: N }
   │
   ├─ Return 200 + JSON response

5. Browser receives
   └─ Parse JSON, update Zustand store
      └─ React re-renders ProductsView
         with fresh data

6. Error handling
   ├─ 401 Unauthorized → Redirect to login
   ├─ 403 Forbidden → Show permission denied
   ├─ 500 Server Error → Show error toast

PDF Generation Flow

1. User clicks "Download Invoice PDF"
   └─ InvoicesView.tsx calls api.pdf(invoiceId)

2. Hono handler: GET /api/invoices/:id/pdf
   ├─ Verify ownership (user's company)
   ├─ Query invoice + related data:
   │  ├─ invoice row
   │  ├─ customer details
   │  ├─ order items
   │  └─ payment status
   ├─ Generate PDF via PDFKit:
   │  ├─ Add header with logo/company info
   │  ├─ Add invoice details table
   │  ├─ Calculate totals
   │  ├─ Generate QR code (invoice verification)
   │  ├─ Set Sarabun Thai font for Thai text
   │  └─ Return PDF buffer
   └─ Send as Content-Type: application/pdf

3. Browser downloads
   └─ Filename: invoice-20240409-001.pdf

Audit Trail

Every CREATE, UPDATE, DELETE operation logs to audit_logs with user, table, record ID, action, and before/after changes. Boss-only visibility via GET /api/audit.

Common Issues & Solutions

API returns 401 Unauthorized

  • Token expired: Log out, log back in (JWT has 7-day expiry)
  • Missing Authorization header: Check api.ts apiFetch() includes Bearer token
  • JWT signature invalid: Check process.env.JWT_SECRET matches on API

Database file grows too large

  • Vacuum: sqlite3 ~/.config/tconsiam/db/tconsiam.db "VACUUM;"
  • Check WAL files: ls -la ~/.config/tconsiam/db/ (should be .db, -wal, -shm)
  • Audit logs huge: Archive old logs or delete (boss-only)

PDF generation fails

  • Font not found: Check api/assets/fonts/Sarabun-Regular.ttf exists
  • Memory error: PDFs can be large; check Node heap size
  • QR code error: Ensure qr.ts library is installed

Nginx returns 502 Bad Gateway

  • API not running: systemctl status tconsiam-api → restart if dead
  • Port 3457 in use: lsof -i :3457 → kill process or restart
  • Nginx config error: nginx -t → check syntax

React component not rendering

  • Hash routing issue: Check URL includes #/dashboard
  • API fetch failing: Check browser Console for errors + network tab
  • Permission denied: User role (boss/warehouse) may not have access

Performance & Scaling Notes

Current Limits (Single VPS, 4GB RAM)

  • Concurrent Users: ~10-20 (SQLite WAL handles this)
  • Database Size: Tested to 500MB+ (no issues)
  • API Response Time: 50-200ms (typical)
  • UI Load Time: 1-2 seconds (first load), instant after cache

Optimization Tips

  • Database Indexes: Already on productId, customerId, orderId in schema
  • Frontend Caching: Vite outputs immutable hashed assets (browser cache forever)
  • API Pagination: All list endpoints support limit/offset
  • Computed Fields: Avoid real-time calculations in loops — use DB aggregates

When to Migrate (PostgreSQL + dedicated DB server)

  • 50+ concurrent users
  • Multiple instances/load balancing needed
  • Advanced replication/failover
  • Complex analytical queries
